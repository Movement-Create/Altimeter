# Altimeter Observability

How to see what the agent actually did, in two tiers: a **short story** you can read after every run, and a **detailed trace** you can drop into when something breaks.

> **Philosophy:** deploy it → play with it → read the story of what happened → if anything failed, drop into the detail logs and debug to the line.

## The two tiers

| Tier | File | Format | When to read it |
|---|---|---|---|
| **Story** | [sessions/&lt;id&gt;.summary.md](sessions/) | Markdown | *Every* run. Tells you "what happened, was it OK?" in 10 seconds. |
| **Detail** | [sessions/&lt;id&gt;.jsonl](sessions/) | JSONL span events | When the story says ❌. Full OpenTelemetry-shaped span tree with durations, token counts, tool inputs, errors, stack traces. |

Both tiers are derived from **one event stream**. There is no separate audit log, no separate metrics pipeline, no chance of drift. Everything the agent does emits a `span_start` / `span_end` event into the session JSONL, and the summary is a deterministic reducer over those events.

A third, **live tier** — the coloured stderr output you see while the agent runs — is also subscribed to the same span stream, so what you watch live is exactly what lands in the file.

## The story tier — `summary.md`

Generated automatically at the end of every `altimeter run` and every `altimeter chat` session. Example:

```markdown
# Session Summary ❌

- **Session:** `2a6a453b-9b4d-49d4-a25b-bcd16e1e12df`
- **Outcome:** failure ❌
- **Turns:** 3
- **LLM calls:** 3
- **Model(s):** qwen3:4b
- **Tokens:** 7344 (in 6867 / out 477)
- **Cost:** $0.0000
- **Wall clock:** 11.13s
- **Reflection fired:** yes

## Tools

| Tool      | Calls | Errors | Total time |
| ---       | ---   | ---    | ---        |
| file_read | 1     | 1      | 3ms        |

## Errors

- **tool.execute (file_read)** — File not found: C:\nonexistent\file.txt
```

**What it tells you:**
- **Outcome** — ✅ ok / ❌ failure / ❔ unknown
- **Turns & LLM calls** — how hard the loop worked
- **Tokens & cost** — so you know what it spent
- **Wall clock** — end-to-end duration
- **Tools table** — which tools ran, how many failed, where time went
- **Errors** — the first line of every failure, tagged by which span raised it
- **Subagents / reflection** — whether either fired

It's generated by [src/observability/summarize.ts](src/observability/summarize.ts). **No LLM call.** It's a pure reducer — runs in milliseconds, deterministic, no flakiness. An optional LLM-narrated paragraph (via `observability.narrate: true`) is planned but not in v1.

**To print the story to stdout after a run:**

```bash
altimeter run "..." --summary
```

## The detail tier — span tree in `sessions/<id>.jsonl`

Every interesting moment in the agent loop becomes a **span** — a named, timed, parented record. Spans follow OpenTelemetry's vocabulary (`trace_id`, `span_id`, `parent_span_id`, `start_time`, `end_time`, `duration_ms`, `status`, `attributes`, `error`) so if you already read OTel traces, this will feel familiar.

### Span taxonomy

```
agent.session
└── agent.turn                         (one per loop iteration)
    ├── llm.call                       (the provider.complete call)
    ├── hook.pre                       (PreToolUse hooks for this tool)
    ├── tool.execute                   (the actual tool dispatch)
    ├── hook.post                      (PostToolUse hooks)
    ├── hook.stop                      (fired once when the turn is text-only)
    └── subagent.run                   (only if the agent tool was called)
        └── agent.session              (child session — joins parent's trace_id)
            └── ... (same structure recursively)

agent.reflection                       (separate root, only if triggered)
└── agent.session
    └── ...
```

Each span carries attributes relevant to its kind:

| Span | Key attributes |
|---|---|
| `agent.session` | `session_id`, `model`, `provider`, `permission_mode`, `max_turns`, `subagent_depth`, end-state: `turns`, `input_tokens`, `output_tokens`, `cost_usd`, `stop_reason` |
| `agent.turn` | `turn_number`, `has_tool_calls`, `tool_call_count`, `tool_error_count`, `exit` |
| `llm.call` | `provider`, `model`, `tool_count`, `input_tokens`, `output_tokens`, `cost_usd`, `stop_reason`, `retries`, `tool_calls_requested` |
| `tool.execute` | `tool_name`, `tool_input` (truncated), `permission_level`, `is_error`, `error_reason` (`unknown_tool` / `not_allowed` / `permission_denied` / `invalid_input` / `tool_error`), `error_message`, `output_length` |
| `hook.pre` / `hook.post` / `hook.stop` | `turn`, `tool_name`, `action` (`allow` / `block` / `modify`), `block_reason?` |
| `subagent.run` | `child_session_id`, `depth`, `prompt_preview`, `child_turns`, `child_cost_usd`, `child_input_tokens`, `child_output_tokens`, `child_stop_reason` |
| `agent.reflection` | `trigger_reason` (`turns>=5` / `tool_error`), `original_turns`, `reflection_turns`, `reflection_cost_usd` |

### Reading the trace

The JSONL file is append-only — one event per line. Filter to just the spans with `jq`:

```bash
jq -c 'select(.type=="span_end") | .data.span | {name, status, duration_ms, attributes}' \
   sessions/<id>.jsonl
```

Render it as a tree (one-off Node script):

```bash
node -e "
const fs = require('fs');
const spans = fs.readFileSync('sessions/<id>.jsonl','utf-8').trim().split('\n')
  .map(JSON.parse).filter(e => e.type==='span_end').map(e => e.data.span);
const print = (s, i=0) => {
  const icon = s.status === 'error' ? '✗' : '✓';
  console.log('  '.repeat(i) + icon + ' ' + s.name + ' (' + s.duration_ms + 'ms)');
  spans.filter(c => c.parent_span_id === s.span_id).forEach(c => print(c, i+1));
};
spans.filter(s => !s.parent_span_id).forEach(s => print(s));
"
```

Example output:

```
✓ agent.session (9359ms)
  ✓ agent.turn (7141ms)
    ✓ llm.call (7133ms)           ← 2127→293 tokens
    ✓ hook.pre (0ms)
    ✓ tool.execute (2ms)          ← file_read
    ✓ hook.post (0ms)
  ✓ agent.turn (2214ms)
    ✓ llm.call (2211ms)           ← 2874→81 tokens
    ✓ hook.stop (0ms)
```

When something fails, the offending span has `status: "error"`, an `error` block with `type`/`message`/`stack`, and — for tool errors — `error_reason` and `error_message` attributes so you can classify the failure without parsing the stack.

## The live tier — `LivePrinter`

Real-time stderr output while the agent runs. Subscribed to the same `tracer.events` EventEmitter that writes the JSONL, so **what you see is what you'll read later**.

Example:

```
[llm] 2127→293 toks · $0.0000 · 7.13s
[tool] file_read
  {"path":"package.json"}
[tool done] file_read · 2ms
[llm] 2874→81 toks · $0.0000 · 2.21s
[summary] sessions/76e9860f-...summary.md
```

On error, a red `[error:tool:file_read] File not found: ...` line is printed as the span ends — regardless of whether the full trace would have been truncated.

Two verbosity levels:
- **`full`** — every tool call, LLM call, subagent, reflection (default)
- **`summary`** — errors only + end-of-session stats (useful in CI or headless scheduled runs)
- **`off`** — silent (no spans written at all — `--no-trace`)

## Configuration

In `altimeter.json5`:

```json5
{
  observability: {
    enabled: true,            // default true
    level: "full",            // "off" | "summary" | "full"
    narrate: false,           // LLM-narrated paragraph in summary.md (v1: unused)
    otel_endpoint: null,      // reserved for a future OTel exporter
  }
}
```

Environment variables:

| Var | Effect |
|---|---|
| `ALTIMETER_OBS_LEVEL` | `off` / `summary` / `full` |
| `ALTIMETER_OBS_NARRATE` | `1` or `true` to enable narration (future) |
| `ALTIMETER_OTEL_ENDPOINT` | Reserved for OTel exporter |

CLI flags (on `altimeter run`):

| Flag | Effect |
|---|---|
| `--summary` | Print the generated `summary.md` to stdout after the run |
| `--no-trace` | Disable spans entirely (equivalent to `ALTIMETER_OBS_LEVEL=off`) |

## How it hooks into the loop

No scattered `console.log` calls. One `Tracer` module ([src/observability/tracer.ts](src/observability/tracer.ts)) with four methods: `startRootSpan`, `startSpan`, `withSpan`, `end`. Every instrumentation point uses one of these.

| Location | Span started | Why here |
|---|---|---|
| [src/core/agent-loop.ts](src/core/agent-loop.ts) | `agent.session` | Wraps the entire `runAgent` call — this is the trace root |
| [src/core/agent-loop.ts](src/core/agent-loop.ts) | `agent.turn` | One per while-loop iteration; body runs inside `tracer.runInSpan` so nested spans inherit it as parent |
| [src/core/agent-loop.ts](src/core/agent-loop.ts) | `llm.call` | Wraps `provider.complete` (via `withRetry`). Stamps tokens & cost on the span after the call returns |
| [src/tools/registry.ts](src/tools/registry.ts) | `tool.execute` | Wraps the whole `executeTool` dispatch including permission checks and Zod validation. `error_reason` distinguishes *why* a tool failed |
| [src/hooks/engine.ts](src/hooks/engine.ts) | `hook.pre` / `hook.post` / `hook.stop` | One per hook-engine fire method. `action` attribute tells you whether a hook blocked, modified, or allowed |
| [src/tools/agent.ts](src/tools/agent.ts) | `subagent.run` | Wraps the recursive `runAgent` call. Passes `_parent_trace_id` / `_parent_span_id` to the child so its `agent.session` span joins the parent trace |
| [src/core/reflection.ts](src/core/reflection.ts) | `agent.reflection` | Wraps the conditional second `runAgent` call. `trigger_reason` is stamped at start |

**Parent linkage.** A top-level `runAgent` call creates a brand-new `trace_id`. When the `agent` tool spawns a subagent, it pulls `tracer.currentContext()` and passes `_parent_trace_id` / `_parent_span_id` into the child `runAgent`, which then joins that trace instead of starting a new one. So parent and all descendants (recursive subagents included) share one `trace_id` and form one tree when you render it. Reflection, by contrast, lives in its own trace — it runs *after* the primary session ended, so linking it to the parent trace would require remembering state we no longer have. The reducer correlates the two by `session_id` instead.

**AsyncLocalStorage.** The tracer uses `AsyncLocalStorage` to infer `parent_span_id` automatically. When you call `tracer.startSpan("tool.execute", …)` from inside a turn body, it reads the current ALS context and fills in the turn's span ID as the parent. This is why the turn body runs inside `tracer.runInSpan(turnSpan, …)` — without that context push, everything would bind to the root.

**Never crashes the agent.** All tracer writes are inside `try { } catch { }`. If the disk fills up, if the JSON serializer trips on something exotic, if `appendEvent` throws — the span is dropped and execution continues. Observability must not become a new failure mode.

## Common questions

**Why not OpenTelemetry SDK?**
Adding `@opentelemetry/api` + `sdk-node` would pull ~5 packages and add startup cost without buying anything for single-user local debugging. The *vocabulary* is OTel-compatible, so the span shape already matches what a future exporter would emit. When you want to ship spans to Jaeger / Honeycomb / Tempo, write a small `exportSpan(span)` function in [src/observability/tracer.ts](src/observability/tracer.ts) that converts our `Span` to `@opentelemetry/api`'s `ReadableSpan` and call it from `emitEnd`. That's the whole port. See the `otel_endpoint` config field (currently unused) for where to wire it.

**Why JSONL and not a real trace backend?**
Spans live next to the data they describe — the session they came from. One file per session, grep-friendly, git-diffable, tail-followable, backup-trivial. For personal and small-team use, that's better than operating a trace backend. For production multi-user deployments, add the OTel exporter above and keep the JSONL as a local backup.

**Why no separate audit log?**
The old [src/security/audit.ts](src/security/audit.ts) used to write a separate `audit-YYYY-MM-DD.jsonl` when `ALTIMETER_AUDIT=1`. Every fact it recorded is now a span attribute on `tool.execute` or `hook.*` in the session JSONL — one source of truth. The `auditLogger` export is kept as a no-op shim for any external code that imports it.

**Resume an old pre-observability session?**
Still works. [src/core/session.ts](src/core/session.ts) `resumeSession` only reads `session_start` / `user_message` / `assistant_message` events; unknown event types (including `span_start` / `span_end`) are silently ignored. Old sessions replay cleanly into a new observability-enabled process.

**Where's the cost ledger?**
`sessions/cost-ledger.jsonl` is still written by [src/core/cost-tracker.ts](src/core/cost-tracker.ts) for cross-session aggregation (total spend per day, per model, etc.). For *single-session* cost, the `llm.call` span's `cost_usd` attribute is the source of truth; for cross-session, use the ledger.
